# PresenceGuard — Development Guide

PresenceGuard automatically sets the Microsoft Teams presence status via
Home Assistant + Microsoft Graph (see [`presenceguard/README.md`](presenceguard/README.md)).

## Commit conventions (Conventional Commits)

All commits and PR titles follow [Conventional Commits](https://www.conventionalcommits.org/):

| Prefix | Semver effect | When |
|--------|---------------|------|
| `feat: ...` | minor bump | New user-visible feature (automation, blueprint, REST command) |
| `fix: ...` | patch bump | Bugfix or behavior correction |
| `chore: ...` | no bump | Tooling, CI, dependencies |
| `docs: ...` | no bump | README, CHANGELOG, comments |
| `refactor: ...` | no bump | Rework without behavior change |
| `feat!: ...` / `BREAKING CHANGE` | major bump | Removes/renames existing inputs or services |

## Release workflow

Releases run automatically via [release-please](https://github.com/googleapis/release-please):

1. Merge `feat:` / `fix:` commits into `main`
2. release-please automatically opens a **release PR** (updates `CHANGELOG.md` + version files)
3. Review and merge the release PR
4. GitHub release + tag are created

Do not edit version strings manually — the workflow handles that.

## CI

| Workflow | Purpose |
|----------|-------|
| `.github/workflows/validate.yaml` | YAML syntax check of all files under `presenceguard/` (incl. blueprint, schedule, rest_commands) on every push/PR |
| `.github/workflows/validate-integration.yaml` | `hassfest` (HA manifest/structure) + `hacs/action` (HACS compatibility) for `custom_components/presenceguard/` |
| `.github/workflows/gitleaks.yml` | Secret scan |
| `.github/workflows/pr-title.yml` | Conventional Commits check of the PR title |
| `.github/workflows/release-please.yml` | Automated release on push to `main` |

## Branch strategy

- `main` — stable
- Feature branches — one branch per feature, PR into `main`
- Branch naming: `feat/<topic>` or names generated by Claude Code

## Two approaches

PresenceGuard comes in two variants (use one per account):

- **A) Custom integration** under `custom_components/presenceguard/` — UI-native
  OAuth login, automatic token renewal, reauth in *Repairs*. HACS-compatible
  (`hacs.json`). See `custom_components/presenceguard/README.md`.
- **B) Classic** under `presenceguard/` — blueprint + REST/shell commands +
  `token_setup.sh`/`token_refresh.sh` (bash/YAML).

## Key files

### Custom integration (`custom_components/presenceguard/`)

| File | Purpose |
|-------|-------|
| `manifest.json` / `hacs.json` (repo root) | Integration and HACS metadata |
| `application_credentials.py` / `config_flow.py` | OAuth2 login + reauth in the HA UI |
| `api.py` / `coordinator.py` | Graph client (`/me`) + poll/reauth detection |
| `__init__.py` | Services `set_offline` / `clear_presence` / `set_presence` |
| `binary_sensor.py` | `binary_sensor.presenceguard_token` (connection status) |

### Classic (HA content under `presenceguard/`)

| File | Purpose |
|-------|-------|
| `presenceguard/blueprints/automation/presenceguard/presence_schedule.yaml` | Blueprint with UI configuration (schedule helper + status dropdown) |
| `presenceguard/schedule_helper_presenceguard.yaml` | Example schedule helper (multiple from/to windows) |
| `presenceguard/rest_commands.yaml` | Graph REST Commands (`set_teams_offline`, `clear_teams_presence`, `set_teams_presence`) |
| `presenceguard/automations_presenceguard.yaml` | Classic, hard-wired automations |
| `presenceguard/command_line_presenceguard.yaml` | Token sensor |
| `presenceguard/template_presenceguard.yaml` | Status sensor `binary_sensor.presenceguard_token` (UI: token data present?) |
| `presenceguard/shell_commands.yaml` | Token refresh call |
| `presenceguard/setup_presenceguard.sh` | Interactive setup wizard (end-to-end setup) |

## HA Custom Tags

The HA configuration YAML uses `!input`, `!include` and `!secret`. These are
registered as `yaml.customTags` in [`.vscode/settings.json`](.vscode/settings.json)
and are treated as pass-through by the validation workflow.
